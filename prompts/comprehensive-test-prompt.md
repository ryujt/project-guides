# 통합 테스트 지시 프롬프트

## 목적

대상 코드/기능에 대해 **가능한 모든 시나리오의 테스트**를 작성하고 실행하여, **오류가 더 이상 발견되지 않을 때까지** 테스트와 수정을 반복한다.
단위 테스트와 E2E 테스트를 함께 사용하고, 발견된 모든 오류와 처리 결과는 **마크다운 이력 문서**로 남긴다.
반복 과정에서 이전에 발견된 오류가 재발하지 않도록 **회귀 테스트**를 항상 함께 수행한다.

## 테스트 범위와 시나리오

가능한 모든 시나리오를 빠짐없이 도출해서 테스트를 작성해줘.

* **정상 흐름(Happy Path)** — 기대한 입력으로 기대한 결과가 나오는 기본 경로.
* **경계 조건(Boundary)** — 최소·최대값, 빈 값, 0, 길이 한계, 오버플로 등 경계에서의 동작.
* **예외·오류 흐름(Error Path)** — 잘못된 입력, 권한 없음, 타임아웃, 네트워크 실패, 예외 발생 시의 처리.
* **상태 전이(State)** — 로딩→성공/실패, 로그인/로그아웃, 편집 모드 토글 등 상태 변화별 동작.
* **동시성·순서(Concurrency)** — 중복 요청, 경쟁 조건, 순서가 뒤바뀐 이벤트 처리.
* **데이터 정합성(Integrity)** — 입력/저장/조회 결과가 일관되는지, 부분 실패 시 롤백 여부.
* **보안·권한(Security)** — 인증·인가, 입력 검증, 주입 공격 방어, 민감 정보 노출 여부.
* **사용자 상황(Persona)** — 신규/기존 사용자, 권한·역할별 사용자, 모바일/데스크톱 등 다양한 사용자의 상황을 상정해 실제 사용 흐름에서 발생할 수 있는 오류를 찾는다.

시나리오를 도출할 때 요구사항, 코드, 기존 테스트를 근거로 삼고, 임의로 범위를 축소하지 마.
누락 가능성이 있는 시나리오는 조용히 생략하지 말고 목록으로 남긴 뒤 판단해줘.

## 테스트 종류

* **단위 테스트** — 함수·모듈 단위의 로직을 격리해서 검증한다. 외부 의존성은 목/스텁으로 대체한다.
* **E2E 테스트** — 실제 사용자 흐름을 처음부터 끝까지 실행해서 검증한다. UI 작업은 실행 화면을 관찰하고, API/백엔드 작업은 실제 응답·로그·상태 변화를 확인한다.
  * 도출한 **가능한 모든 시나리오에 대해 E2E 테스트를 진행**한다. 일부 시나리오만 골라서 실행하지 않는다.
  * **다양한 사용자의 상황을 상정해** 사이트(애플리케이션)를 실제로 탐색하며 오류를 찾고, 발견된 오류는 수정한다.
* 두 종류 모두에서 실패 케이스와 경계 조건을 반드시 포함한다.
* 버그를 수정할 때는 **재현 테스트를 먼저 작성**한 뒤 수정한다.

## 테스트-수정 반복 루프

문제가 없을 때까지, 즉 오류가 더 이상 발견되지 않을 때까지 아래 순서를 반복해서 수행해줘.

1. 대상 코드/기능의 현재 동작과 요구사항을 파악한다.
2. 위 범위에 따라 시나리오 목록을 작성한다.
3. 시나리오별로 단위 테스트와 E2E 테스트를 작성한다.
4. **이력 문서에 기록된 과거 오류의 회귀 테스트를 함께 포함한다.**
5. 전체 테스트를 실행한다.
6. 실패한 항목이 있으면 원인을 분석하고 최소 변경으로 수정한다.
7. 수정 후 전체 테스트(신규 + 회귀)를 다시 실행한다.
8. 새로 발견된 오류는 이력 문서에 추가하고, 그에 대한 회귀 테스트를 만든다.
9. 아래 종료 조건을 만족할 때까지 3~8을 반복한다.

### 회귀 방지 원칙

* 매 반복마다 **이전에 발견된 모든 오류의 테스트를 반드시 다시 실행**한다.
* 한 번 통과한 테스트라도 이후 수정으로 깨질 수 있으므로 항상 전체를 함께 돌린다.
* 이력 문서에 있는 오류는 대응하는 테스트가 반드시 하나 이상 존재하도록 유지한다.

## 오류 이력 문서

발견된 오류와 처리 결과를 마크다운 문서로 남겨줘.

* 파일 위치와 이름을 명확히 정하고(예: `test-history/오류-이력.md`), 반복 내내 같은 문서에 누적한다.
* 오류 하나당 아래 항목을 기록한다.

  | 항목 | 설명 |
  | --- | --- |
  | ID | 오류 식별자 (예: `BUG-001`) |
  | 발견 일시 | 발견한 날짜/반복 회차 |
  | 시나리오 | 어떤 테스트/흐름에서 발견했는지 |
  | 증상 | 관찰된 잘못된 동작 |
  | 원인 | 근본 원인 분석 |
  | 수정 내용 | 변경한 파일과 핵심 수정 |
  | 회귀 테스트 | 재발 방지를 위해 추가한 테스트 |
  | 상태 | 발견 / 수정 중 / 해결 / 보류 |

* 각 반복 회차마다 실행한 테스트 명령, 통과/실패 요약을 함께 남긴다.

## 종료 조건

다음 조건을 모두 만족하면 작업을 종료해줘.

* 도출한 모든 시나리오에 대한 테스트가 작성되고 실행됨.
* 단위 테스트와 E2E 테스트가 모두 통과함.
* 이력 문서에 기록된 모든 오류가 해결되고, 각 오류에 대한 회귀 테스트가 통과함.
* 연속 반복에서 새로운 오류가 더 이상 발견되지 않음.
* 보류한 이슈가 있다면 그 사유와 후속 계획이 명확히 기록됨.

## 프롬프트 문서 갱신

테스트 작업이 완료되면, **이번 작업에서 새롭게 발견된 오류 유형을 이 프롬프트 문서에 추가해줘.**

* 아래 「발견된 오류 유형 카탈로그」 섹션에, 이번에 처음 마주친 오류의 **패턴**을 항목으로 추가한다.
* 특정 프로젝트에 종속된 상세가 아니라, 다음 테스트에서도 미리 점검할 수 있는 **일반화된 형태**로 적는다.
* 이미 카탈로그에 있는 유형이면 중복 추가하지 말고, 필요 시 설명만 보강한다.
* 이 카탈로그는 다음 테스트에서 시나리오를 도출할 때 체크리스트로 함께 활용한다.

## 멀티 에이전트 협업

여러 전문 에이전트(**Architect · Critic · Developer · Tester**)에게 역할을 분담시켜, 상호 견제와 반복 검토 루프로 테스트 품질을 끌어올릴 수 있다.
공통 행동 원칙과 반복 검토 루프, 종료 조건은 [`./multi-agent-task-prompt.md`](./multi-agent-task-prompt.md) 를 함께 따른다.
이 작업에서는 **Tester Agent**가 시나리오 도출·테스트 작성·실행·이력 관리를 주도하고, **Critic Agent**가 누락된 시나리오와 불충분한 검증을 끊임없이 지적한다.

## 발견된 오류 유형 카탈로그

다음 패턴은 이후 종합 테스트에서도 선제적으로 점검한다.

* **낙관적 비동기 처리와 authoritative 상태의 경쟁** — 전송 함수가 서버 ACK나 버전 갱신 전에 성공 처리되어 연속 변경이 같은 stale version을 사용하는지 확인한다. 빠른 연속 클릭, 서로 다른 권한의 동시 변경, 충돌 응답 뒤 UI 대기 상태를 함께 검증한다.
* **성공한 부수 효과와 응답 유실 사이의 복구 경쟁** — 서버 처리는 성공했지만 응답만 유실된 상황에서 일반 흐름과 자동 복구가 동시에 실행되어 중복 생성·결제·정산을 일으키지 않는지 확인한다. 최초 서버 ID와 재시도 결과의 동일성도 검증한다.
* **비동기 로딩 중 disabled trigger의 포커스 유실** — 버튼 클릭 직후 비활성화되고 나중에 dialog가 열릴 때 `document.activeElement`가 이미 `body`로 이동할 수 있다. 명시적 복귀 target, focus trap, Escape 종료 후 복귀를 실제 브라우저에서 확인한다.
* **엄격한 CSP와 런타임 WASM/동적 코드 생성의 비호환** — 단위 테스트에서는 동작하지만 실제 `script-src` 정책에서 WebAssembly compile 또는 eval이 차단되는지 확인한다. CSP를 불필요하게 완화하지 않고 대체 구현의 최대 입력 메모리 경계도 검증한다.
* **오류 후 재마운트로 인한 사용자 입력 상태 유실** — 인증·비밀번호·네트워크 오류가 gate나 form을 재마운트해 표시 이름 등 재사용 가능한 입력까지 초기화하지 않는지 확인한다. 비밀번호 같은 민감 입력은 보존하지 않는 정책과 구분한다.
* **재연결 성공 뒤 남는 stale 오류 UI** — 연결 상태는 정상으로 복구됐지만 이전 연결 오류 banner나 pending 표시가 남지 않는지 확인한다. 연결 계열 오류만 제거하고 파일 업로드 등 무관한 오류는 보존해야 한다.
* **DOM 순서 기반 반응형 CSS의 기능 은닉** — `:first-child`, `:nth-child()` 같은 위치 선택자가 DOM 변경 후 종료·초기화 등 핵심 동작을 모바일에서 숨기지 않는지 실제 viewport와 접근성 트리에서 확인한다.
* **테스트 base URL과 서버 Origin allowlist 불일치** — `localhost`와 `127.0.0.1`처럼 기능상 같은 로컬 주소라도 WebSocket Origin 검증에서는 다르다. 브라우저 E2E 주소와 허용 Origin을 일치시키고 실시간 연결까지 확인한다.
* **느슨한 test double이 숨기는 브라우저 API 제약** — WebSocket close code, clipboard, media permission처럼 브라우저가 동기 예외나 별도 허용 범위를 적용하는 API는 실제 브라우저와 같은 제약을 fake에도 구현한다. 단위 테스트 통과만으로 브라우저 호환을 판단하지 않는다.
* **재연결 중 stale CONNECTING 상태 고착** — 오프라인 자동 재시도와 online 이벤트가 경쟁할 때 이미 열린 `CONNECTING` 객체를 정상 복구로 오인하지 않는지 확인한다. 제한 시간 안에 열리지 않은 연결은 폐기하고 새 연결로 교체하며, 떠난 peer의 지연 신호 오류가 살아 있는 host 연결 전체를 종료하지 않아야 한다.
* **중계 계층이 보강한 필드와 엄격한 parser의 계약 불일치** — 서버가 보안 라우팅을 위해 `targetPeerId` 같은 필드를 추가할 때 송신·중계·수신의 모든 중복 validator가 같은 optional bounded schema를 사용하는지 검증한다. 한 단계의 exact-key 검사가 정상 메시지를 조용히 폐기하지 않아야 한다.
* **부분 컨테이너 재생성에 취약한 공유 네트워크 네임스페이스** — sidecar의 `network_mode: service:...`나 loopback 의존이 대상 컨테이너만 재생성했을 때 이전 namespace에 남지 않는지 확인한다. 내부 서비스 endpoint와 브라우저 공개/presign endpoint를 분리하고 partial recreate 뒤 실제 I/O를 검증한다.
