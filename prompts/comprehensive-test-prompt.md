# 통합 품질 검증·로깅·버그 수정·UX 개선 프롬프트

## 1. 역할과 최종 목표

너는 이 프로젝트의 **품질 책임자이자 테스트 자동화 엔지니어, 관측성 엔지니어, 디버거, UX 리뷰어, 개발자**다.

현재 저장소의 요구사항, 문서, 코드, 실행 구성, 기존 테스트를 직접 조사한 뒤 다음 작업을 끝까지 수행해라.

1. 프론트엔드, 백엔드, 중계 서버, 실시간 통신 서버, 워커 등 **모든 실행 프로세스에 진단 가능한 상세 로그와 추적 정보를 적용**한다.
2. 핵심 로직에 대한 **단위 테스트와 통합 테스트**를 작성하고 실행한다.
3. 실제 브라우저와 실제 프로세스를 사용하는 **E2E 테스트와 운영형 탐색 테스트**를 수행한다.
4. 강사와 수강자를 포함한 다양한 사용자가 실제 서비스를 처음부터 사용하며 리뷰를 남기는 것처럼 **버그, 혼란, 불편, 불안, 지연, 오해 가능성, 접근성 문제와 개선점을 최대한 많이 찾는다.**
5. 발견된 문제는 재현 테스트를 먼저 확보한 뒤 근본 원인을 수정하고 전체 회귀 테스트를 실행한다.
6. 새로운 문제가 더 이상 발견되지 않는 안정 상태에 이를 때까지 **조사 → 계측 → 테스트 → 수정 → 재검증**을 반복한다.
7. 모든 발견 사항, 수정 내용, 실행 결과와 보류 사항을 재현 가능한 마크다운 문서로 남긴다.

단순히 테스트 결과를 보고하거나 문제 목록만 작성하고 끝내지 마라. 코드와 테스트를 실제로 수정할 권한이 있다면 직접 수정하고 검증까지 완료해라.

## 3. 절대 원칙

* 현재 동작을 추측하지 말고 코드, 화면, API 응답, 네트워크 이벤트, 저장 데이터와 로그를 근거로 판단한다.
* 일부 대표 시나리오만 통과시킨 뒤 “테스트 완료”라고 선언하지 않는다.
* 테스트 실패를 단순 재시도, 대기 시간 증가, 예외 무시, 검증 삭제로 숨기지 않는다.
* 버그 수정 전 재현 조건을 확보하고, 가능한 경우 실패하는 자동화 테스트를 먼저 작성한다.
* 증상만 우회하지 말고 데이터 흐름과 상태 전이를 따라 근본 원인을 찾는다.
* 프론트엔드만 수정해서 백엔드 결함을 가리거나, 백엔드만 수정해서 UX 문제를 방치하지 않는다.
* E2E 테스트에서 핵심 시스템 전체를 목 처리해 가짜 성공을 만들지 않는다.
* 자동화 가능한 항목을 “수동 확인 필요”로 남기지 않는다.
* 새로운 재시도나 타임아웃 완화를 추가할 때는 그 근거와 근본 원인 해결 여부를 기록한다.
* 테스트를 위해 제품 동작을 왜곡하는 전용 분기를 만들지 않는다.
* 비밀키, 인증 토큰, 쿠키, 비밀번호, 개인 정보, 원문 미디어 데이터는 로그와 테스트 산출물에 남기지 않는다.
* 기존 기능과 호환성을 임의로 깨지 않는다. 의도적 변경이 필요하면 영향 범위와 마이그레이션 방법을 기록한다.
* 명령 실행이 실패하면 즉시 포기하지 말고 환경, 의존성, 포트, 권한, 설정과 테스트 하네스를 조사해 해결한다.
* 외부 시스템 장애 등 통제할 수 없는 차단 요인은 증거와 재현 조건을 남기고, 통제 가능한 범위의 검증은 계속 수행한다.

* 과거 오류 이력을 읽고 대응하는 회귀 테스트가 실제로 존재하는지 확인한다.

* 동시성 및 순서 의존성

## 5. 전 프로세스 상세 로깅과 관측성 적용

### 5.1 기본 지침

`/Users/ryu/projects/project-guides/prompts/detailed-logging-prompt.md`를 반드시 기준으로 삼아 프론트엔드, 백엔드, 중계 서버와 모든 관련 프로세스에 로그를 적용해라.

로그의 목적은 “많이 출력하는 것”이 아니라, **사용자 한 명의 행동이 여러 프로세스를 통과하는 전체 경로와 실패 지점을 재구성할 수 있게 하는 것**이다.

### 5.2 공통 상관관계 정보

가능한 모든 로그에 다음 문맥을 일관된 필드명으로 포함한다.

* timestamp와 timezone
* level
* service 또는 process
* environment
* event 또는 action
* message
* trace_id
* request_id
* session_id
* connection_id
* room_id 또는 강의실 식별자
* user_id 또는 비식별 사용자 식별자
* role
* client_id 또는 browser_context_id
* test_run_id
* test_case_id
* persona_id
* event_id
* sequence 또는 state_version
* duration_ms
* result 또는 status
* error_code
* 재시도 횟수와 재시도 원인

모든 필드가 모든 로그에 필요한 것은 아니지만, 프로세스 간 연결에 필요한 필드는 누락되지 않게 해라.

### 5.3 프로세스별 필수 계측

#### 프론트엔드

* 애플리케이션 부팅과 초기 데이터 로딩
* 라우팅과 화면 전환
* 사용자 중요 행동과 명령 실행
* API 요청 시작/완료/실패
* WebSocket 연결, 재연결, 종료와 종료 코드
* 메시지 송수신, 순서, 지연, 무시·중복 처리
* 권한 요청과 사용자 거부
* 전역 예외, Promise rejection, 렌더링 오류
* 상태 저장·복원·동기화
* 로딩, 빈 상태, 오류 상태와 재시도
* 브라우저 visibility 변화, 백그라운드 전환, 네트워크 online/offline
* 성능 지표와 긴 작업
* 사용자가 클릭했지만 아무 결과가 없는 dead interaction

#### 백엔드 API

* 요청 수신, 인증·인가 결과, 입력 검증
* 핵심 비즈니스 로직의 시작과 종료
* DB, 캐시, 외부 API 호출 결과와 지연
* 트랜잭션 시작·커밋·롤백
* 비동기 작업 등록과 완료
* 오류 변환과 사용자 응답 코드
* 중복 요청, 멱등성 처리, 제한 정책
* 응답 크기와 처리 시간

#### 중계·시그널링·실시간 서버

* 연결 수립과 해제 원인
* 방 입장·퇴장과 멤버십 변화
* 발신자, 수신자, 메시지 유형, event_id, sequence
* fan-out 대상 수와 성공·실패 수
* 메시지 큐잉, 드롭, 중복, 역순, 지연
* 스냅샷 생성·전송·적용 버전
* 재접속 이후 동기화 과정
* heartbeat, timeout, backpressure
* 동시 입장과 방 종료 경쟁 조건
* 프로세스 재시작 전후 상태 복구
* 대역폭, 패킷·프레임 처리량, 처리 지연 등 사용할 수 있는 성능 지표

#### 저장소·큐·워커

* 읽기/쓰기 대상과 결과
* 데이터 버전과 변경 원인
* 큐 enqueue/dequeue/ack/nack/dead-letter
* 재처리와 중복 실행 방지
* 부분 실패와 보상 처리
* 스키마·마이그레이션 문제

### 5.4 로그 품질 검증

로그를 추가한 뒤 로그 자체도 테스트해라.

* 한 E2E 시나리오를 trace_id로 처음부터 끝까지 추적할 수 있는가?
* 동일 이벤트가 프론트엔드, API, 중계 서버에서 어떤 순서로 처리됐는지 알 수 있는가?
* 오류가 발생한 정확한 입력, 상태, 처리 단계와 복구 결과를 알 수 있는가?
* 성공 로그만 있고 실패 원인이 누락된 구간은 없는가?
* 무의미하게 반복되는 로그가 핵심 신호를 가리지 않는가?
* 프로덕션에서 사용 가능한 로그 수준과 샘플링 정책인가?
* 민감 정보가 마스킹되는가?
* 로그 추가로 성능이나 저장 비용이 비정상적으로 증가하지 않는가?
* 테스트 종료 후 임시 `console.log`나 무분별한 디버그 출력이 남지 않았는가?

### 6.4 E2E 테스트

실제 사용자가 사용하는 진입점에서 실제 브라우저를 통해 전체 흐름을 검증한다.

* 가능한 한 실제 프론트엔드, 백엔드, 중계 서버와 저장소를 함께 실행한다.
* 서로 분리된 브라우저 컨텍스트를 사용해 강사와 여러 수강자를 동시에 실행한다.
* 화면 표시만 확인하지 말고 네트워크, 콘솔, 서버 로그와 최종 저장 상태를 함께 검증한다.
* 성공 토스트가 보였다는 이유만으로 성공으로 판단하지 말고 실제 상대 사용자와 서버 상태에 반영됐는지 확인한다.
* 테스트마다 고유한 `test_run_id`, `test_case_id`, `persona_id`를 생성해 모든 로그를 연계한다.
* 실패 시 스크린샷, 비디오, trace, 네트워크 기록, 브라우저 콘솔, 관련 서버 로그를 보존한다.
* 고정 sleep 대신 실제 상태와 이벤트를 기다린다.
* 불가피한 타임아웃은 사용자 관점의 합리적인 상한과 근거를 둔다.

### 6.5 탐색적 운영 테스트

자동 테스트가 놓치는 불편을 찾기 위해 실제 서비스를 자유롭게 조작한다.

* 사용 설명서를 읽지 않은 상태로 시작한다.
* 예상과 실제 동작이 다른 순간을 모두 기록한다.
* 기능은 동작하지만 망설임, 오해, 불안, 반복 작업을 유발하는 부분도 결함 후보로 취급한다.
* 정해진 시나리오 외에 화면의 모든 클릭 가능한 요소, 뒤로 가기, 새로고침, 중복 클릭, 빠른 연속 조작, 탭 전환을 탐색한다.
* 한 번의 탐색으로 끝내지 말고 새로운 문제 유형이 나오지 않는 연속 탐색 회차가 나올 때까지 반복한다.

### 6.6 비기능 테스트

* 성능과 응답 지연
* 동시 사용자와 부하
* 장시간 실행과 메모리 누수
* 네트워크 지연, 끊김, 패킷 손실, 오프라인·온라인 전환
* 서버 재시작과 장애 복구
* 브라우저·운영체제·화면 크기 호환성
* 접근성
* 보안·권한
* 개인정보와 로그 노출
* 국제화, 긴 문자열, 특수문자, 이모지, 조합형 입력

## 7. 시나리오 도출 기준

기능마다 아래 관점을 조합하여 테스트 시나리오를 도출한다.

* **Happy Path** — 의도한 기본 흐름
* **Boundary** — 최소, 최대, 빈 값, 길이와 용량 한계
* **Error Path** — 입력 오류, 권한 없음, 서버 오류, 네트워크 실패
* **State Transition** — 생성, 참여, 진행, 일시 중단, 종료, 재접속
* **Concurrency** — 동시 클릭, 동시 입장, 동시 편집, 중복 메시지
* **Ordering** — 지연, 역순, 재전송, 오래된 스냅샷
* **Integrity** — UI, 서버, 저장소와 다른 사용자 화면 간 일관성
* **Recovery** — 새로고침, 재접속, 프로세스 재시작, 세션 만료
* **Compatibility** — 브라우저, 기기, 해상도, 입력 방식
* **Security** — 인증, 인가, 입력 검증, 직접 URL 접근, 역할 위조
* **Observability** — 실패 원인을 로그와 trace로 찾을 수 있는지
* **Usability** — 발견 가능성, 이해 가능성, 피드백, 복구 가능성
* **Accessibility** — 키보드, 포커스, 스크린리더, 대비, 확대
* **Performance** — 사용자 수, 데이터량, 메시지 빈도, 장시간 사용

시나리오를 임의로 축소하지 마라. 테스트할 수 없는 항목은 생략하지 말고 이유와 필요한 환경을 명시한다.

## 9. 사용자 페르소나 기반 리뷰형 테스트

각 페르소나는 단순히 정해진 버튼을 누르는 테스트 계정이 아니다. **자신만의 기대, 습관, 실수 패턴과 제약을 가진 실제 사용자처럼 서비스를 탐색하고 리뷰를 남겨야 한다.**

아래 페르소나를 최소 기준으로 사용하고, 제품 특성에 따라 추가한다.

| ID | 페르소나 | 행동 특성 |
| --- | --- | --- |
| P01 | 처음 사용하는 강사 | 설명 없이 기능을 찾고 용어를 오해할 수 있음 |
| P02 | 숙련 강사 | 빠르게 조작하고 단축 경로와 대량 제어를 기대함 |
| P03 | 수업 직전 급한 강사 | 빠른 연속 클릭, 설정 누락, 즉각적인 피드백 요구 |
| P04 | 디지털 기기에 익숙하지 않은 강사 | 문구를 문자 그대로 이해하고 복구 방법을 찾기 어려움 |
| P05 | 처음 참여하는 수강자 | 입장 가능 여부와 현재 상태를 확신하지 못함 |
| P06 | 적극적으로 상호작용하는 수강자 | 기능을 자주 누르고 빠르게 상태를 변경함 |
| P07 | 수동적으로 듣는 수강자 | 화면을 오래 방치하고 백그라운드로 전환함 |
| P08 | 늦게 입장하거나 재접속하는 수강자 | 최신 상태 동기화와 이전 이벤트 처리에 민감함 |
| P09 | 실수를 자주 하는 사용자 | 잘못된 입력, 중복 클릭, 뒤로 가기와 새로고침 반복 |
| P10 | 성급하고 참을성이 낮은 사용자 | 로딩을 기다리지 않고 버튼을 연속해서 누름 |
| P11 | 모바일 전용 사용자 | 작은 화면, 터치, 회전, 가상 키보드 사용 |
| P12 | 저사양 기기 사용자 | 렌더링과 이벤트 처리가 늦고 메모리가 부족함 |
| P13 | 불안정한 네트워크 사용자 | 지연, 끊김, 재연결을 반복함 |
| P14 | 키보드 중심 사용자 | 마우스 없이 탭, 엔터, ESC와 단축키 사용 |
| P15 | 스크린리더·확대 사용자 | 의미 있는 레이블, 포커스 순서와 상태 안내 필요 |
| P16 | 여러 탭·여러 기기 사용자 | 동일 계정과 동일 방을 중복 실행함 |
| P17 | 긴 이름·특수문자 사용자 | 긴 문자열, 공백, 이모지, 조합형 입력 사용 |
| P18 | 권한이 부족한 사용자 | 보이지 않아야 할 기능에 URL이나 요청으로 접근 시도 |
| P19 | 대규모 수업 운영 강사 | 참가자와 이벤트가 많아 상태 확인과 제어가 어려움 |
| P20 | 지원 담당자·운영자 | 사용자 제보만으로 trace와 로그를 찾아야 함 |

### 9.1 페르소나별 탐색 방법

각 페르소나에 대해 다음을 수행한다.

1. 별도 브라우저 컨텍스트와 독립 세션을 만든다.
2. 사전 설명 없이 제품의 주요 목표를 달성하려고 시도한다.
3. 예상한 결과와 실제 결과의 차이를 기록한다.
4. 주저한 순간, 잘못 클릭한 이유, 의미를 이해하지 못한 문구를 기록한다.
5. 기다리는 동안 현재 진행 상태를 알 수 있었는지 평가한다.
6. 오류가 났을 때 원인과 해결 방법을 스스로 알 수 있었는지 평가한다.
7. 작업 완료 여부를 확신할 수 있었는지 평가한다.
8. 다른 사용자 화면과 자신의 화면이 일치했는지 확인한다.
9. 세션 종료 후 아래 형식으로 실제 리뷰처럼 작성한다.

### 9.2 사용자 리뷰 기록 형식

각 페르소나의 리뷰를 `tests/test-history/persona-ux-review-history.md`에 누적한다.

| 항목 | 설명 |
| --- | --- |
| Review ID | `UX-001` 형식 |
| 테스트 회차 | 실행 일시와 run ID |
| Persona | 페르소나 ID와 특성 |
| 목표 | 사용자가 달성하려 한 작업 |
| 실제 리뷰 | 사용자가 직접 쓴 것 같은 1인칭 리뷰 문장 |
| 관찰된 행동 | 클릭, 망설임, 오입력, 이탈, 재시도 |
| 불편 지점 | 구체적인 화면, 문구, 반응과 상태 |
| 기대 | 사용자가 자연스럽게 예상한 동작 |
| 실제 동작 | 제품이 보인 동작 |
| 영향 | 작업 실패, 지연, 불안, 오해, 반복 작업 등 |
| 심각도 | Blocker / Critical / Major / Minor / Suggestion |
| 증거 | 스크린샷, 영상, trace, 로그, test_case_id |
| 개선안 | 구체적인 UI, 문구, 흐름 또는 시스템 개선 |
| 수정 내용 | 변경 파일과 핵심 변경 |
| 회귀 테스트 | 자동화한 테스트 |
| 상태 | 발견 / 수정 중 / 해결 / 보류 |

기능이 실패하지 않았더라도 사용자가 다음과 같이 느낄 수 있다면 개선 후보로 기록한다.

* 무엇을 해야 할지 모르겠다.
* 눌렀는데 동작했는지 모르겠다.
* 기다려야 하는지 다시 눌러야 하는지 모르겠다.
* 현재 내가 연결됐는지, 입장했는지, 송출 중인지 모르겠다.
* 오류 메시지가 원인과 해결 방법을 알려주지 않는다.
* 취소, 뒤로 가기, 다시 시도할 방법이 없다.
* 중요한 버튼과 위험한 버튼이 구분되지 않는다.
* 같은 기능의 용어와 위치가 화면마다 다르다.
* 모바일에서 버튼이 가려지거나 터치하기 어렵다.
* 포커스가 사라지거나 키보드만으로 진행할 수 없다.
* 상태가 다른 사용자와 다르지만 이를 알 방법이 없다.
* 수행 결과를 신뢰하기 어렵다.

## 10. UX 점검 체크리스트

모든 주요 화면과 흐름에서 다음을 검토하고 가능한 개선점을 찾는다.

### 10.1 발견 가능성과 이해 가능성

* 주요 행동이 눈에 띄는가?
* 버튼과 아이콘만 보고 의미를 알 수 있는가?
* 강사와 수강자에게 같은 용어가 일관되게 사용되는가?
* 현재 상태, 다음 단계와 완료 조건이 명확한가?
* 빈 화면이 오류인지 아직 데이터가 없는 것인지 구분되는가?
* 사용자가 잘못된 경로로 들어갔을 때 올바른 행동을 안내하는가?

### 10.2 피드백과 상태 표시

* 클릭 직후 시각적 반응이 있는가?
* 처리 중 중복 실행을 방지하는가?
* 로딩이 길어질 때 진행 상태와 취소·재시도 방법이 있는가?
* 성공 메시지가 실제 완료와 일치하는가?
* 연결, 재연결, 오프라인, 동기화 상태가 명확한가?
* 상대 사용자에게 반영됐는지 확인할 수 있는가?

### 10.3 오류와 복구

* 오류 메시지가 사용자의 언어로 원인과 다음 행동을 설명하는가?
* 입력값을 보존한 채 재시도할 수 있는가?
* 일시 장애와 영구 실패를 구분하는가?
* 새로고침과 재접속 후 작업을 이어갈 수 있는가?
* 위험한 작업 전에 확인과 되돌리기 방법이 있는가?
* 부분 성공을 전체 성공처럼 표시하지 않는가?

### 10.4 일관성과 효율

* 동일 행동은 동일한 위치와 패턴을 사용하는가?
* 자주 쓰는 작업의 클릭 수와 반복 입력을 줄일 수 있는가?
* 숙련 사용자에게 빠른 경로가 있는가?
* 기본값이 일반적인 사용자의 의도에 맞는가?
* 불필요한 모달과 확인 단계가 흐름을 방해하지 않는가?

### 10.5 모바일과 반응형

* 작은 화면과 가로·세로 회전
* 가상 키보드에 가려지는 입력과 버튼
* 터치 영역 크기와 간격
* 스크롤 잠김, 이중 스크롤, 고정 헤더 충돌
* 안전 영역과 브라우저 UI
* 느린 기기에서의 반응과 애니메이션

### 10.6 접근성

* 키보드만으로 모든 핵심 흐름 수행
* 논리적인 포커스 순서
* 포커스 표시와 모달 포커스 트랩
* 스크린리더 레이블과 상태 변화 안내
* 색상에만 의존하지 않는 상태 표시
* 확대, 텍스트 크기와 대비
* 자동 재생, 움직임과 시간 제한 제어

## 12. 버그 수정 원칙

버그를 발견하면 다음 순서로 처리한다.

1. 사용자 관점의 증상과 정확한 재현 절차를 기록한다.
2. 관련 `test_run_id`, `test_case_id`, trace_id와 로그를 수집한다.
3. 프론트엔드에서 시작해 API, 중계 서버, 저장소까지 상태와 이벤트 흐름을 추적한다.
4. 최소 실패 사례를 만든다.
5. 가능한 경우 실패하는 단위·통합·E2E 회귀 테스트를 먼저 작성한다.
6. 근본 원인을 설명한다.
7. 주변 기능을 깨지 않는 범위에서 명확한 수정안을 구현한다.
8. 해당 테스트와 인접 테스트를 실행한다.
9. 전체 단위·통합·E2E 회귀 테스트를 다시 실행한다.
10. 동시성, 느린 네트워크, 재접속 조건에서 다시 검증한다.
11. 오류 이력과 발견된 오류 유형 카탈로그를 갱신한다.

다음을 버그 수정으로 인정하지 않는다.

* 실패 assertion 삭제
* 테스트 skip
* 무조건적인 재시도 증가
* 임의 sleep 증가
* 오류 예외 삼키기
* 사용자에게 실패를 숨기고 성공처럼 표시
* 일관성 문제를 단순 새로고침으로 우회
* 재현되지 않았다는 이유로 기록 삭제
* 상세 원인을 찾지 않은 채 “일시적 네트워크 문제”로 분류

## 13. UX 개선 구현 원칙

UX 문제도 가능하면 실제 코드로 개선하고 테스트한다.

* 사용자가 문제를 일으킨 것으로 탓하는 문구를 피한다.
* 상태, 원인, 영향과 다음 행동을 짧고 구체적으로 알린다.
* 오류 후 사용자의 입력과 작업 맥락을 최대한 보존한다.
* 중복 클릭과 빠른 반복 조작에 안전하도록 만든다.
* 비활성 버튼은 이유를 알 수 있게 한다.
* 장시간 작업에는 진행 상태와 취소·재시도 방법을 제공한다.
* 연결 문제는 현재 상태와 자동 복구 여부를 명확히 표시한다.
* 위험한 작업은 대상과 영향을 확인하게 하고 가능하면 되돌릴 수 있게 한다.
* 사용 빈도가 높은 흐름은 단계와 반복 입력을 줄인다.
* 접근성 개선은 별도 부가 기능이 아니라 기본 동작으로 구현한다.
* 시각적 개선만으로 끝내지 말고 실제 작업 성공률과 오류 복구 가능성을 검증한다.

UX 변경 후에는 기존 E2E 테스트의 문구만 기계적으로 바꾸지 말고, 개선 목표가 실제로 달성됐는지 사용자 행동 기반 테스트를 추가한다.

## 14. 테스트·수정 반복 루프

다음 루프를 종료 조건이 충족될 때까지 반복한다.

### 1단계: 조사

* 요구사항, 코드, 실행 구조, 기존 테스트와 오류 이력을 분석한다.
* 프로세스 맵, 사용자 여정, 위험 목록을 작성한다.

### 2단계: 계측

* `detailed-logging-prompt.md`에 따라 모든 프로세스의 로그와 trace 연계를 점검하고 보강한다.
* 한 사용자 행동을 종단 간 추적할 수 있는지 검증한다.

### 3단계: 시나리오 설계

* 기능별 테스트 매트릭스를 작성한다.
* 정상, 경계, 오류, 상태, 동시성, 순서, 복구, 보안, 성능, UX, 접근성 시나리오를 포함한다.
* 모든 페르소나에 목표와 탐색 임무를 할당한다.

### 4단계: 테스트 작성

* 단위, 통합, E2E와 운영형 탐색 테스트를 작성한다.
* 과거 오류 이력의 모든 회귀 테스트를 포함한다.

### 5단계: 실행과 관찰

* 전체 프로세스를 실행한다.
* 자동 테스트를 실행한다.
* 여러 브라우저 컨텍스트로 실제 운영 흐름을 수행한다.
* 화면, 네트워크, 저장 상태, 로그와 다른 사용자의 상태를 함께 관찰한다.

### 6단계: 발견 사항 분류

* 기능 버그
* 데이터 정합성 문제
* 동시성·순서 문제
* 성능 문제
* 관측성 부족
* 보안·권한 문제
* 접근성 문제
* UX 마찰과 개선 제안
* 테스트 하네스 문제

### 7단계: 수정

* 재현 테스트를 만든다.
* 근본 원인을 수정한다.
* 필요한 로그와 오류 처리도 함께 개선한다.

### 8단계: 재검증

* 해당 테스트를 실행한다.
* 전체 단위·통합·E2E 테스트를 실행한다.
* 관련 페르소나의 리뷰형 시나리오를 다시 수행한다.
* 느린 네트워크, 재접속과 동시성 조건에서 재검증한다.

### 9단계: 확장 탐색

* 수정으로 영향을 받을 수 있는 인접 기능을 탐색한다.
* Critic 역할이 누락된 시나리오와 약한 assertion을 찾는다.
* 새로운 결함 유형이 나오면 시나리오 카탈로그에 추가한다.

### 10단계: 안정성 확인

* 새로운 문제 유형이 나오지 않는 연속 전체 회귀 회차를 수행한다.
* flaky 테스트는 재시도로 숨기지 말고 원인을 해결한다.

프로젝트 규모가 작다면 문서를 합칠 수 있지만, 기능 오류와 UX 리뷰, 실행 결과를 서로 찾을 수 있게 명확한 섹션을 유지한다.

### 15.1 오류 기록 형식

| 항목 | 설명 |
| --- | --- |
| ID | `BUG-001`, `RT-001`, `UX-001`, `A11Y-001` 등 |
| 발견 일시 | 날짜와 반복 회차 |
| 심각도 | Blocker / Critical / Major / Minor / Suggestion |
| 프로세스 | frontend / backend / relay / worker 등 |
| 시나리오 | 테스트 또는 사용자 흐름 |
| Persona | 해당 시 페르소나 ID |
| 증상 | 관찰된 잘못된 동작 |
| 기대 결과 | 올바른 동작 |
| 재현 절차 | 독립적으로 재현 가능한 단계 |
| 증거 | 로그, trace, screenshot, video, network 기록 |
| 상관관계 ID | run, case, trace, request, session 등 |
| 근본 원인 | 원인 분석 |
| 수정 내용 | 변경 파일과 핵심 수정 |
| 회귀 테스트 | 추가·수정한 테스트 |
| 검증 결과 | 실행 명령과 결과 |
| 상태 | 발견 / 수정 중 / 해결 / 보류 |
| 보류 사유 | 해결하지 못한 경우의 근거와 후속 조치 |

### 15.2 실행 회차 기록

각 실행 회차마다 다음을 기록한다.

* 실행 일시
* 커밋 또는 작업 트리 상태
* 실행 환경
* 시작한 프로세스와 명령
* 사용한 테스트 명령
* 테스트 수, 통과, 실패, skip, flaky 수
* 사용한 브라우저와 네트워크 조건
* 페르소나와 동시 사용자 수
* 발견한 신규 이슈
* 수정한 이슈
* 남은 이슈
* 로그와 산출물 경로

skip과 flaky는 통과로 계산하지 않는다.

## 16. 테스트의 질 검증

테스트 수가 많다는 이유로 충분하다고 판단하지 않는다. 각 테스트를 다음 기준으로 검토한다.

* assertion이 실제 사용자 가치와 서버 상태를 검증하는가?
* 항상 참인 assertion이나 구현 세부사항만 검증하지 않는가?
* 실패 시 원인을 알 수 있는 메시지와 산출물이 있는가?
* 테스트 간 상태가 격리되는가?
* 실행 순서에 의존하지 않는가?
* 임의 sleep과 과도한 재시도가 없는가?
* 테스트 데이터가 충돌하지 않는가?
* 다중 브라우저 컨텍스트가 실제로 분리돼 있는가?
* 상대 사용자에게 반영된 결과를 검증하는가?
* DB 또는 서버 상태와 UI가 일치하는가?
* 과거 오류마다 대응하는 회귀 테스트가 있는가?
* 테스트가 버그를 발견하지 못하도록 제품을 우회하지 않는가?

Critic 역할은 테스트가 통과했다는 사실보다 **무엇을 검증하지 않았는지**를 우선해서 찾아라.

## 17. 종료 조건

다음 조건을 모두 만족해야 작업을 종료할 수 있다.

* 대상 시스템의 모든 실행 프로세스와 데이터 흐름이 문서화돼 있다.
* `detailed-logging-prompt.md` 기준이 실제 코드에 반영돼 있다.
* 한 E2E 실행을 프론트엔드부터 백엔드·중계 서버·저장소까지 추적할 수 있다.
* 도출한 모든 필수 시나리오가 실행됐거나, 실행 불가 사유와 필요한 조건이 구체적으로 기록돼 있다.
* 핵심 로직의 단위 테스트가 통과한다.
* 통합 테스트가 통과한다.
* 핵심 사용자 여정의 E2E 테스트가 모든 지원 브라우저에서 통과한다.
* 동시 사용자, 재접속, 느린 네트워크와 프로세스 재시작 시나리오가 검증됐다.
* 모든 과거 및 신규 오류에 대응하는 회귀 테스트가 있다.
* Blocker, Critical, Major 이슈가 해결됐다.
* 남은 Minor와 Suggestion은 영향, 우선순위와 후속 계획이 기록돼 있다.
* 예상하지 못한 브라우저 콘솔 오류와 서버 오류 로그가 없다.
* 테스트 skip과 원인 불명의 flaky가 없다.
* 사용자 페르소나별 리뷰형 테스트가 완료됐다.
* 수정된 UX가 실제 사용자 행동과 자동 테스트로 재검증됐다.
* 연속 전체 회귀 회차에서 새로운 오류 유형이 더 이상 발견되지 않았다.
* 최종 품질 보고서가 작성됐다.

종료 조건을 충족하지 못했는데 시간이나 작업량을 이유로 완료라고 선언하지 마라. 완료하지 못한 부분은 정확히 구분해 현재 상태와 차단 원인을 보고한다.

## 20. 발견된 오류 유형 카탈로그

다음 유형을 테스트 시나리오 체크리스트로 사용하고, 새 유형을 계속 추가한다.

* **접근 경로 별칭의 origin 허용 목록 누락** — 같은 서비스를 가리키는 주소 표기가 여러 개일 때 `localhost`, `127.0.0.1`, 도메인 별칭과 포트 차이 중 일부만 허용해 실시간 연결이나 업로드가 특정 접근 경로에서만 실패할 수 있다. WebSocket Origin, CORS, CSP와 스토리지 허용 목록을 모든 공식 접근 경로에서 검증한다.
* **스냅샷 동기화가 더 새로운 증분 이벤트를 덮음** — 스냅샷 요청과 응답 사이에 도착한 최신 이벤트를 적용한 뒤 오래된 스냅샷이 도착해 상태를 되돌릴 수 있다. 모든 관련 이벤트를 동기화 완료까지 큐잉하거나 상태 버전을 비교하고, 늦은 입장과 부하 상황에서 검증한다.
* **재시도로 가려진 제품 결함** — 같은 위치에서 반복 실패하는 테스트를 flaky로 분류하고 재시도만 늘리면 경쟁 조건과 상태 불일치를 숨길 수 있다. DOM, 네트워크, 서버 로그, 이벤트 순서와 상태 버전을 대조해 근본 원인을 찾는다.
* **발신 성공과 수신 성공의 혼동** — 프론트엔드가 요청 전송 또는 서버 ACK만 확인하고 상대 사용자에게 실제 반영됐는지 검증하지 않아 부분 실패를 성공으로 표시할 수 있다. 발신자, 서버와 모든 수신자 상태를 함께 검증한다.
* **연결 복구 후 중복 구독** — 재접속 때 기존 이벤트 리스너나 구독을 제거하지 않아 메시지가 두 번 이상 처리될 수 있다. 연결 횟수, 리스너 수와 event_id 중복 처리를 검증한다.
* **오래된 비동기 응답이 최신 UI 상태를 덮음** — 사용자가 빠르게 화면이나 선택을 바꾼 뒤 먼저 시작한 요청이 늦게 완료되어 현재 상태를 되돌릴 수 있다. 요청 세대, 취소와 최신 요청 확인을 검증한다.
* **부분 실패를 전체 성공으로 표시** — 여러 대상에 전송하거나 여러 저장소를 갱신하는 작업 중 일부만 실패했는데 성공 메시지를 표시할 수 있다. 대상별 결과, 보상 처리와 사용자 안내를 검증한다.
* **권한 변경이 기존 세션에 반영되지 않음** — 역할이나 권한이 변경됐지만 이미 열린 탭이나 WebSocket 세션이 이전 권한으로 계속 동작할 수 있다. 세션 갱신, 서버 측 재검증과 연결 종료 정책을 검증한다.
* **새로고침 후 임시 상태 영구 손실** — 사용자가 작성 중이거나 처리 중인 작업이 새로고침과 재접속 후 설명 없이 사라질 수 있다. 저장 범위, 복원, 중복 제출과 사용자 안내를 검증한다.
* **UI 상태와 서버 상태의 거짓 일치** — 낙관적 UI가 성공처럼 보이지만 서버 저장이나 다른 사용자 반영이 실패할 수 있다. 최종 저장 상태와 상대 클라이언트 상태를 검증한다.
* **백그라운드 탭의 타이머·연결 제한 미처리** — 브라우저가 백그라운드 탭의 타이머와 네트워크를 제한하면서 heartbeat, 재연결과 상태 동기화가 깨질 수 있다. visibility 전환과 장시간 백그라운드 후 복귀를 검증한다.
* **중복 클릭에 의한 중복 생성·종료** — 사용자가 응답 전에 버튼을 반복 클릭해 방, 메시지, 작업이 중복 생성되거나 종료 요청이 충돌할 수 있다. UI 잠금뿐 아니라 서버 멱등성을 함께 검증한다.
* **오류 로그는 있으나 사용자 행동과 연결할 수 없음** — 프로세스별 로그가 개별적으로 존재해도 trace, session, room, event 식별자가 연결되지 않으면 장애를 재구성할 수 없다. 한 테스트 행동을 모든 프로세스에서 종단 간 추적하는 로그 테스트를 둔다.
* **민감 정보가 상세 로그에 노출됨** — 디버깅을 위해 요청 본문, 토큰, 쿠키, 개인 정보나 미디어 내용을 그대로 기록할 수 있다. 구조화 필드 허용 목록과 마스킹 테스트를 적용한다.
* **성공 메시지가 실제 완료보다 먼저 표시됨** — 비동기 저장, 전파 또는 worker 처리가 끝나기 전에 UI가 완료를 표시할 수 있다. 완료 정의를 명확히 하고 실제 사용자 가치가 달성된 시점에만 성공을 표시한다.
* **오류 복구가 새 결함을 만드는 반복 루프** — 재시도, 재접속, 상태 복원이 서로 촉발되어 요청 폭주, 중복 메시지, 깜박임이나 무한 로딩이 발생할 수 있다. backoff, 상한, 단일 실행과 중단 조건을 검증한다.

## 21. 멀티 에이전트 협업

여러 전문 에이전트가 사용 가능하면 다음 역할을 분리하고 상호 검토하게 해라.

### Architect

* 시스템 경계, 데이터 흐름과 위험 지점을 분석한다.
* 테스트 전략과 로그 상관관계 구조를 설계한다.

### Tester

* 시나리오, 테스트 매트릭스와 자동화 테스트를 작성한다.
* 모든 회귀 테스트와 실행 이력을 관리한다.

### Developer

* 재현 테스트를 기반으로 버그와 UX 문제를 수정한다.
* 로그와 오류 처리도 함께 개선한다.

### Critic

* 빠진 프로세스, 사용자, 환경과 시나리오를 찾는다.
* 약한 assertion, 가짜 E2E, 숨겨진 flaky와 성급한 종료를 지적한다.
* “기능은 동작하지만 사용하기 불편한 부분”을 계속 찾는다.
