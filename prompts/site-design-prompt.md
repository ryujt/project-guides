# 사이트 설계 프롬프트

## 목적

주어진 요구사항에 맞는 **사이트 설계 문서**를 작성한다.
설계는 **페이지 이동(네비게이션) 중심의 시나리오**로 풀어내고, 각 페이지의 화면 레이아웃과 프로세스 흐름(상태도)까지 함께 문서화한다.
여러 전문 에이전트가 역할을 나눠 상호 견제하며, 유사 서비스 벤치마킹을 근거로 설계 품질을 끌어올린다.

## 네비게이션 중심 설계

`/Users/ryu/projects/project-guides/guides/navigation-diagram-guide.md` 의 표기법과 작성 규칙을 반드시 따라서, **페이지 이동 중심의 시나리오별 설계 문서**를 작성한다.

* **전체 흐름 요약으로 시작한다.**
  * 사이트 전체를 관통하는 간략한 화면 이동 흐름을 먼저 하나의 다이어그램으로 제시한다.
  * 독자가 사이트의 큰 그림을 먼저 잡을 수 있게 한다.

* **전체 다이어그램 이후, 독립적인 시나리오별 상세 설명을 이어서 작성한다.**
  * 각 시나리오는 사용자의 목적(예: 회원가입, 검색, 결제, 설정 변경 등)을 기준으로 분리한다.
  * 각 시나리오는 제목 + 상세 설명(트리거 조건, 관련 화면/API, 주요 분기·예외) + 다이어그램으로 구성한다.
  * 정상 흐름뿐 아니라 에러·검증 실패·취소 등 예외 흐름도 포함한다.
  * 각 코드 블록은 ```` ```navigation ```` 언어 태그로 감싼다.

* **모든 다이어그램 밑에는 짧은 문장의 나열로 간명한 설명을 덧붙인다.**
  * 긴 문단이 아니라, 화면 이동의 핵심을 짧은 문장으로 나열한다.
  * 시나리오의 트리거 조건, 주요 분기, 관련 API를 명시한다.

* **Page와 Component를 구분한다.**
  * Page는 주소(라우트)를 가지는 단위, Component는 주소 없이 Page에 포함되는 단위다.
  * 페이지 이동은 없지만 중요 컴포넌트가 바뀌는 경우(예: 탭 전환)에는 가이드의 괄호 표기법을 사용한다.

## 화면 레이아웃 설계

`/Users/ryu/projects/project-guides/guides/screen-layout-guide.md` 의 `layout` DSL 표기법을 따라서, **각 페이지마다 화면 레이아웃**을 설계한다.

* 네비게이션 다이어그램에 등장한 각 Page마다 레이아웃을 하나씩 정의한다.
* `V`(세로) / `>`(가로) 연산자와 비율 지정 문법을 사용한다.
* 각 코드 블록은 ```` ```layout ```` 언어 태그로 감싼다.
* 레이아웃 밑에도 짧은 문장으로 각 영역의 역할을 설명한다.

## 프로세스 흐름(상태도) 설계

`/Users/ryu/projects/project-guides/guides/state-diagram-guide.md` 의 표기법을 따라서, **각 페이지 및 중요 컴포넌트마다 상태도로 프로세스 흐름**을 설명한다.

* 화면 이동 없이 내부에서 벌어지는 상태 변화(로딩→성공/실패, 입력 검증, 모달 열림/닫힘, 편집 모드 토글 등)를 다룬다.
* 다른 페이지로의 전환은 상태도가 아니라 네비게이션 다이어그램에서 다룬다.
* 상태 변화가 거의 없는 단순 안내 화면은 스킵하고, 스킵 사유를 짧게 남긴다.
* 각 코드 블록은 ```` ```state ```` 언어 태그로 감싼다.

## 벤치마킹

* 요구사항과 유사한 사이트/서비스를 검색해서 벤치마킹할 수 있는 자료를 찾아 참고한다.
* 참고한 서비스의 화면 구성·네비게이션 패턴·핵심 흐름 중 이번 설계에 반영한 부분을 명시한다.
* 단순 모방이 아니라, 이번 요구사항에 맞게 취사선택한 근거를 함께 남긴다.

## 멀티 에이전트 협업

여러 명의 전문 에이전트(**Architect · Critic · Developer · Tester**)에게 역할을 분담시켜 상호 견제와 반복 검토 루프로 설계를 완성한다.
공통 행동 원칙과 반복 검토 루프, 종료 조건은 [`./multi-agent-task-prompt.md`](./multi-agent-task-prompt.md) 를 함께 따른다.

이 작업에서 각 에이전트의 관점은 아래와 같이 특화한다.

### A. Architect Agent

* **임무**: 요구사항 해석, 페이지/컴포넌트 구조 정의, 네비게이션 시나리오 설계, 화면 레이아웃과 상태도 초안 작성.
* **원칙**: 사용자 흐름의 자연스러움, 화면 구성의 일관성, 확장 가능성.
* **산출물**: 전체 흐름 요약, 시나리오 목록, 페이지 목록, 각 페이지의 레이아웃·상태도 초안, 명시적 가정.

### B. Critic Agent

* **임무**: 설계의 개선점과 문제점을 **끊임없이** 찾아 피드백한다.
* **검토 항목**:
  * 누락된 시나리오·진입점·예외 흐름
  * 화면 이동이 아닌 내용이 네비게이션에 섞였는지(가이드 안티패턴 위반)
  * Page/Component 구분 오류, 불필요한 컴포넌트 전이 표기
  * 레이아웃과 시나리오의 불일치
  * 상태도에 화면 이동이 섞였는지
  * 벤치마킹 근거 부족, 사용성 저하 지점
* **원칙**: "완벽한 설계는 없다." 모든 가정에 의문을 제기한다.
* **산출물**: 반드시 고쳐야 할 문제 / 고치면 좋은 문제 / 수용 가능한 트레이드오프.

### C. Developer Agent

* **임무**: 승인된 설계를 각 가이드 표기법에 맞는 실제 다이어그램 코드(`navigation` / `layout` / `state`)로 정확히 작성한다.
* **원칙**: 가이드 문법 준수, 표기 일관성, 추측 금지(요구사항·벤치마킹 근거 기반).
* **산출물**: 완성된 시나리오/레이아웃/상태도 블록, 각 블록 하단의 짧은 설명.

### D. Tester Agent

* **임무**: 설계 문서의 정합성을 검증한다.
* **검증 항목**:
  * 네비게이션에 등장한 모든 Page가 레이아웃과 상태도로 커버되는지
  * 다이어그램 문법이 각 가이드에 맞는지(노드 명명 규칙, 분기 표기 등)
  * 시나리오 간 화면 이동이 서로 모순되지 않는지
  * 진입점에서 모든 주요 화면에 도달 가능한지
* **산출물**: 검증 항목별 통과/실패, 발견된 불일치, 재검토 요청.

## 산출물

하나의 마크다운 설계 문서로 작성하며, 아래 순서를 따른다.

1. **개요** — 요구사항 요약, 설계 범위, 명시적 가정.
2. **벤치마킹** — 참고 서비스와 이번 설계에 반영한 점.
3. **전체 흐름 요약** — 사이트 전체 네비게이션 다이어그램 + 짧은 설명.
4. **시나리오별 네비게이션 상세** — 전체 흐름 요약에 이어서, 시나리오마다 제목 + 상세 설명(트리거 조건, 관련 화면/API, 주요 분기·예외) + ```` ```navigation ```` 다이어그램.
5. **페이지별 화면 레이아웃** — 페이지마다 ```` ```layout ```` 다이어그램 + 영역 설명.
6. **페이지·컴포넌트별 상태도** — 대상마다 ```` ```state ```` 다이어그램 + 프로세스 설명.
7. **미해결 이슈 / 다음 단계** — Critic·Tester가 남긴 잔여 지적과 후속 작업.
