# Job Flow 상세 설명서(Walkthrough) 생성 프롬프트

> **jobflow 다이어그램이 포함된 문서를 입력받아, 모든 다이어그램을 비전문가도 읽을 수 있는 '상세 설명서'로 풀어 쓰는** 작업 지시문이다. 이 프롬프트에 **대상 문서를 함께 지정**해 실행한다.
> 원칙: **원본 무수정 · 전체→조각(에피소드) 2단 구성 · 비전문가 언어 · 조각 합집합=전체**.
> jobflow 표기의 의미는 [job-flow-diagram-guide.md](../guides/job-flow-diagram-guide.md)(무시 분기 · `.result` · 분기값)와 [method-R.md](../guides/method-R.md)(`message` 경계 표기)를 따른다 — 여기서 재서술하지 않는다.

## 입력

- **필수**: ```` ```jobflow ```` 펜스가 1개 이상 포함된 마크다운 문서 경로(1개 이상).
- 문서에 jobflow 가 여러 개면 **모든 다이어그램을 각각** 설명한다(누락 금지). 특정 다이어그램만 지정받았으면 그것만.

## 산출물

- 원본 문서와 **같은 폴더**에 `jobflow-walkthrough.md` 1개. 같은 폴더에 여러 원본을 다루면 `{원본파일명}-jobflow-walkthrough.md` 로 구분한다. 이미 있으면 갱신한다.
- 서두 인용블록에 원본 문서 링크와 "이 문서가 무엇을 다루는지" 한 줄을 적는다.
- 다이어그램이 여러 개면 다이어그램마다 최상위 장(`## 다이어그램 N — {사건형 제목}`)으로 나누고, 각 장 안에서 아래 구성을 반복한다.

## 다이어그램별 구성 (순서 고정)

1. **전체 흐름**: 원본 다이어그램 전체를 그대로 인용 + **한 문장 요약**("~하면 → ~하고 → ~한다" 꼴).
2. **등장인물 표**: `Object:` 의 모든 객체에 일상 언어 역할을 붙인다(진행자·심판·창구·중계·보관함 등). 흐름을 지휘하는 orchestrator 를 "주인공"으로 표시한다.
3. **표기 읽는 법**: 그 다이어그램에 **실제 등장하는 표기만** 골라 3~5개를 짧게 설명한다. 등장한다면 반드시 포함할 것 —
   - 화살표 없는 단독 줄(무시 분기): "그 갈림길에서는 아무 일도 일어나지 않는다".
   - `X.message.Y`: 기능을 직접 부르는 것이 아니라 **경계(화면·사람·외부)로 결과를 흘려보내는 전달**.
   - 연속 `.result`: 받은 결과를 변형 없이 다음 구간으로 넘기는 릴레이.
4. **에피소드**: 전체를 의미 있는 작은 사건 단위로 잘라 순서대로 설명한다(아래 규칙).
5. **다시 한 장으로**: 에피소드들을 한 줄씩 번호 목록으로 재요약해 전체 그림으로 되돌아온다.

## 에피소드 분할 규칙

- 한 에피소드 = **하나의 작은 사건**(대략 흐름 1~4줄): 진입, 검증·분기, 요청이 가는 길, 결과가 오는 길, 반영, 후처리 같은 단위.
- 에피소드 조각의 **합집합 = 원본의 모든 흐름 줄**. 누락 금지, 문맥상 필요한 중복 인용은 허용.
- 각 에피소드의 jobflow 펜스는 원본의 `orchestrator:` / `Object:` 헤더를 **그대로 유지**한다 — 어느 조각을 봐도 전체 등장인물 속 위치가 보이게.
- 에피소드 제목은 기술 용어가 아니라 **사건을 서술하는 문장형**으로 쓴다(예: "심판이 반칙을 검사한다", "답이 돌아오는 길").
- 각 에피소드는 펜스 아래 **불릿 2~4개**로 설명한다: 무엇이 일어나는지 → 분기가 있으면 갈래별로 → 사용자 눈에 보이는 결과.

## 작성 규칙

- **원본 무수정**: 다이어그램의 줄을 한 글자도 바꾸지 않는다(추가·삭제·개명 금지). 설명만 붙인다.
- **비전문가 언어**: 짧은 문장, 능동태. 전문용어는 첫 1회만 풀어쓰고 재사용한다. 은유는 등장인물 역할 수준으로 가볍게(과한 비유 금지).
- **사실 범위 준수**: 설명은 원본 문서(와 지정된 근거 코드)의 사실을 벗어나지 않는다. 원본만으로 알 수 없는 동작은 단정하지 않는다.
- 객체 이름은 코드 이름 그대로 쓰되, 첫 등장 시 괄호로 역할을 병기한다 — 예: "심판(**RuleEngine**)".

## 형식 스켈레톤

````markdown
# {사건형 제목} — Job Flow 상세 설명서

> 대상 다이어그램: [원본문서.md §N](./원본문서.md) — {한 줄 소개}

## 전체 흐름

```jobflow
{원본 전체 그대로}
```

한 문장으로 요약하면: **{A하면 → B하고 → C한다}.**

### 등장인물

| 이름 | 쉬운 역할 |
|---|---|
| **{Object}** | {일상 언어 역할} |

### 표기 읽는 법 ({N}가지만 알면 된다)

* {이 다이어그램에 등장하는 표기만}

---

## 1. {사건형 제목}

```jobflow
orchestrator: {원본 헤더 유지}
Object: {원본 헤더 유지}

{해당 에피소드의 원본 줄}
```

* {설명 불릿 2~4개}

## 2. {사건형 제목}
...

---

## 다시 한 장으로

1. **{동사}** — {한 줄 요약}
2. ...
````

## 자가검증 (종료 게이트 — 위반 0까지 수리 후 재검증)

- [ ] 대상 문서의 **모든** ```` ```jobflow ```` 펜스가 설명됐다(지정 범위 기준).
- [ ] 원본 다이어그램 줄이 변형 없이 그대로다(전체 인용 = 원본 diff 0).
- [ ] 에피소드 조각의 합집합이 전체 흐름 줄을 모두 덮는다(누락 0).
- [ ] 모든 에피소드 펜스에 원본 `orchestrator:`/`Object:` 헤더가 유지됐다.
- [ ] 무시 분기·`message`·`.result` 릴레이 등 특수 표기가 등장하는 곳마다 설명이 붙었다.
- [ ] 비전문가 기준 충족: 등장인물 표 · 문장형 에피소드 제목 · 짧은 문장 · 전문용어 1회 풀어쓰기.
- [ ] 마지막에 "다시 한 장으로" 재요약이 있다.
